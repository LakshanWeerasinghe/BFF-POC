# SonicWave Backend — Specification

## Overview

A Ballerina REST API that serves as the data layer for the SonicWave webapp. It manages a song library and handles authentication using JWT tokens. The webapp's Express server proxies all `/api/*` requests to this backend.

---

## Tech Stack

| Concern | Choice |
|---|---|
| Database | H2 (file-based, persistent across restarts) |
| Port | `8080` |

---

## Database


### Table Schemas

#### `users`

Stores authenticated users. Users are auto-created on first login (no pre-registration required).

```sql
CREATE TABLE users (
    id         BIGINT       GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    username   VARCHAR(100) NOT NULL UNIQUE,
    created_at TIMESTAMP    DEFAULT CURRENT_TIMESTAMP
);
```

#### `songs`

Stores the song library entries.

```sql
CREATE TABLE songs (
    id         BIGINT       GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    title      VARCHAR(255) NOT NULL,
    artist     VARCHAR(255) NOT NULL,
    album      VARCHAR(255) NOT NULL DEFAULT 'Unknown Album',
    duration   VARCHAR(20)  NOT NULL DEFAULT '0:00',
    cover_url  VARCHAR(500),
    created_at TIMESTAMP    DEFAULT CURRENT_TIMESTAMP
);
```

---

### Initial Seed Data

On application startup, if the `songs` table is **empty**, populate it with the following records:

| title | artist | album | duration | cover_url |
|---|---|---|---|---|
| Midnight City | M83 | Hurry Up, We're Dreaming | 4:03 | https://picsum.photos/seed/m83/400/400 |
| Starboy | The Weeknd | Starboy | 3:50 | https://picsum.photos/seed/starboy/400/400 |
| Blinding Lights | The Weeknd | After Hours | 3:20 | https://picsum.photos/seed/blinding/400/400 |

This seeding must be idempotent — it must not insert duplicates if the server is restarted with an existing populated database. Check `songs` table row count before inserting.

Implementation: use a Spring `ApplicationRunner` or `CommandLineRunner` bean, or a `data.sql` file guarded by a count check.

## Authentication

### Strategy

Username-only login (no password). This matches the existing frontend behaviour. When a user logs in:
1. If the username does not exist in `users`, create a new record.
2. Generate a signed JWT token containing the `username` as the subject.
3. Return the token and user object to the caller.

All endpoints except `POST /api/login` require a valid JWT in the `Authorization: Bearer <token>` header.

## API Endpoints

### Base URL

```
http://localhost:8080
```

---

### `POST /api/login`

Authenticate a user by username. Creates the user if they do not already exist.

**Request**

```
Content-Type: application/json
```

```json
{
  "username": "musiclover99"
}
```

**Validation**
- `username` is required and must not be blank.

**Response — 200 OK**

```json
{
  "token": "<jwt-token>",
  "user": {
    "username": "musiclover99"
  }
}
```

**Response — 400 Bad Request** (username missing or blank)

```json
{
  "error": "Username is required"
}
```

---

### `GET /api/songs`

Return the full list of songs in the library.

**Request Headers**

```
Authorization: Bearer <jwt-token>
```

**Response — 200 OK**

```json
[
  {
    "id": "1",
    "title": "Midnight City",
    "artist": "M83",
    "album": "Hurry Up, We're Dreaming",
    "duration": "4:03",
    "coverUrl": "https://picsum.photos/seed/m83/400/400"
  },
  ...
]
```

**Response — 401 Unauthorized** (missing or invalid token)

```json
{
  "error": "Unauthorized"
}
```

---

### `GET /api/songs/{id}`

Return a single song by its ID.

**Request Headers**

```
Authorization: Bearer <jwt-token>
```

**Response — 200 OK**

```json
{
  "id": "1",
  "title": "Midnight City",
  "artist": "M83",
  "album": "Hurry Up, We're Dreaming",
  "duration": "4:03",
  "coverUrl": "https://picsum.photos/seed/m83/400/400"
}
```

**Response — 404 Not Found**

```json
{
  "error": "Song not found"
}
```

**Response — 401 Unauthorized**

```json
{
  "error": "Unauthorized"
}
```

---

### `POST /api/songs`

Add a new song to the library.

**Request Headers**

```
Authorization: Bearer <jwt-token>
Content-Type: application/json
```

**Request Body**

```json
{
  "title": "Blinding Lights",
  "artist": "The Weeknd",
  "album": "After Hours",
  "duration": "3:20",
  "coverUrl": "https://example.com/cover.jpg"
}
```

**Field Rules**
- `title` — required, not blank.
- `artist` — required, not blank.
- `album` — optional; defaults to `"Unknown Album"` if omitted or blank.
- `duration` — optional; defaults to `"0:00"` if omitted or blank.
- `coverUrl` — optional; if omitted or blank, generate a placeholder: `https://picsum.photos/seed/<timestamp>/400/400`.

**Response — 201 Created**

```json
{
  "id": "4",
  "title": "Blinding Lights",
  "artist": "The Weeknd",
  "album": "After Hours",
  "duration": "3:20",
  "coverUrl": "https://example.com/cover.jpg"
}
```

**Response — 400 Bad Request**

```json
{
  "error": "Title and artist are required"
}
```

**Response — 401 Unauthorized**

```json
{
  "error": "Unauthorized"
}
```

## JSON Response Shape

The frontend expects the song `id` field as a **string** (not a number). The JSON serialiser must convert the `Long` database ID to a `String` in all song responses.

This can be done with a `@JsonSerialize(using = ToStringSerializer.class)` annotation on the `id` field of the Song entity/DTO, or by configuring Jackson globally.

## CORS

Allow requests from the webapp origin so the Express BFF can proxy to this backend, and the frontend can reach it directly during development:

```

Allowed methods: GET, POST, PUT, DELETE, OPTIONS
Allowed headers: Authorization, Content-Type
```
